# 上下文压缩模型测试报告

## 1. 测试背景

为评估轻量化大模型在本地环境下执行上下文压缩任务的可行性与效果，本次测试针对多种 GGUF 量化模型开展统一对比，重点考察以下问题：

- 在 8GB 显存、Windows 11 环境下，各模型是否具备稳定部署和运行能力。
- 在统一提示词与生成参数下，各模型在压缩效率、推理性能和信息保真方面的差异。
- 在中短文本与长上下文场景下，哪类模型更适合作为实际业务中的上下文压缩方案。

本报告面向模型选型和部署决策，侧重给出可复核的测试结果、明确的分析结论和可执行的后续建议。

## 2. 测试结论摘要

本次共测试 5 个本地可部署模型，统一采用 Ollama + GGUF 方案，围绕性能和压缩质量两个维度进行评估。综合测试结果表明：

- Qwen3.5-4B-Q4_K_M 综合表现最佳，在语义相似度、关键信息召回率和整体稳定性方面均处于领先位置，适合作为当前默认上下文压缩模型。
- glm-edge-4b-compress 在首 Token 延迟和吞吐率方面表现最优，适合对响应速度敏感的场景，但在长文本场景下关键信息召回明显偏弱。
- gemma-4-E2B-it-Q8_0 具备一定综合平衡性，可作为备选模型，但在质量和速度上整体仍弱于 Qwen3.5-4B。
- gemma-4-E4B-it-Q4_K_M 在当前测试硬件条件下推理代价偏高（8G的GPU显存不够，部分推理在CPU上），冷启动和总耗时显著落后，不适合作为当前环境下的主力方案。
- Phi-4-mini-instruct-Q4_K_M 压缩最激进，但语义保真和关键信息保留损失较大，适合极限压缩实验，不适合作为正式业务方案。

综合部署成本、推理速度和压缩质量，建议优先采用 Qwen3.5-4B-Q4_K_M 作为默认方案，并将 glm-edge-4b-compress 作为低时延场景的备选方案。

## 3. 测试范围与对象

### 3.1 测试目标

本次测试聚焦上下文压缩任务，目标是比较不同轻量化模型在以下方面的能力：

- 压缩后文本是否保留原始上下文中的关键信息。
- 压缩结果是否具备较好的语义一致性和语言流畅性。
- 模型在本地设备上的冷启动速度、首 Token 响应和整体生成效率。

## 5. 测试方法

### 5.1 统一任务定义

本次测试任务为对会话历史、工具调用结果和决策过程进行上下文压缩，要求模型在显著缩短输出长度的同时，尽可能保留关键信息、决策链路和工具执行结果。

### 5.2 统一提示词与生成参数

所有模型均未进行微调，仅通过统一提示词和生成参数适配文本压缩任务。核心提示词要求模型保留关键数值、标识符、日期、工具结果及因果逻辑，同时删除闲聊、冗余日志和无关说明。

统一生成参数如下：

| 参数 | 设置值 |
| --- | --- |
| num_ctx | 32768（glm为16384，该模型原生上下文窗口为8192） |
| temperature | 0.1 |
| top_p | 0.85 |
| top_k | 40 |
| repeat_penalty | 1.15 |

### 5.3 测试数据

本次共使用 4 组测试数据：

| 测试集 | 数据特点 | 说明 |
| --- | --- | --- |
| test_a | 中短文本、模拟数据 | 由大模型构造，重点考察中英文混合场景下的压缩效果 |
| test_b | 长文本、真实日志 | 基于 QIRA 后台真实日志构建 |
| test_c | 长文本、真实日志 | 基于 QIRA 后台真实日志构建 |
| test_d | 超长文本、真实日志 | 基于 QIRA 后台真实日志构建 |

其中，test_a 更适合观察模型在结构化信息和双语场景下的精细保留能力；test_b、test_c、test_d 更适合观察模型在真实业务日志中的长上下文压缩能力。

### 5.4 评价指标

本次测试从性能和质量两个维度进行评估。

#### 性能指标

| 指标 | 含义 | 判定方向 |
| --- | --- | --- |
| coldstart_time_s | 模型冷启动耗时 | 越低越好 |
| ttft_ms | 首 Token 延迟 | 越低越好 |
| total_time_s | 完整生成耗时 | 越低越好 |
| throughput_token_s | 吞吐率 | 越高越好 |
| output_length | 压缩后文本长度 | 需结合质量指标综合判断 |
| compression_multiple | 原文长度 / 压缩后长度 | 越高表示压缩越激进 |

#### 质量指标

| 指标 | 含义 | 判定方向 |
| --- | --- | --- |
| semantic_similarity | 原文与压缩文本的语义相似度，使用 BGE-M3 评估 | 越高越好 |
| ppl | 压缩文本困惑度，使用 llama-3.1-8B（GGUF）评估 | 越低越好 |
| recall_rate | 压缩文本中的关键信息召回率，使用 Qwen3-Max 以 LLM-as-Judge 方式评估 | 越高越好 |

说明：原始结果中 compression_ratio 存在两种口径。为避免混淆，本报告统一采用以下定义：

- 压缩倍数：原文长度 / 压缩后长度。
- 压缩率：1 - 压缩后长度 / 原文长度。

## 7. 结果分析

### 7.1 性能分析

1. GLM-Edge-4B 在多数测试集上表现出最低的 TTFT 和最高的吞吐率，是本次测试中响应速度最优的模型。
2. Qwen3.5-4B 在速度和质量之间取得了较好平衡，冷启动和整体耗时均明显优于 Gemma 系列。
3. Gemma-4-E4B 在当前测试设备上表现出显著的性能劣势，尤其在冷启动和总耗时方面，不适合频繁调用场景。
4. Phi-4-mini 的输出长度最短，说明压缩激进程度最高，但这种高压缩是以明显牺牲质量为代价实现的。

### 7.2 质量分析

1. 在 test_a 中，Qwen3.5-4B 的召回率达到 0.95，明显优于其他模型，说明其对结构化关键信息、数值与双语内容的保留能力较强。
2. 在 test_b 至 test_d 的长文本真实日志场景中，各模型召回率普遍偏低，说明随着上下文长度上升，模型更倾向于生成概括性摘要，而非精确保留关键字段。
3. Qwen3.5-4B 在长文本场景中仍保持相对领先，说明其作为默认方案更稳妥。
4. GLM-Edge-4B 尽管在速度上有明显优势，但在长文本场景下召回率多次为 0，表明其输出更适合快速概述，不适合高保真压缩。
5. Phi-4-mini 的 PPL 在多个长文本场景中显著偏高，说明其输出在语言自然度和可读性上存在不稳定风险。

### 7.3 业务适配性分析

结合性能与质量结果，可以将各模型的适用场景概括如下：

- Qwen3.5-4B：适用于正式业务场景，尤其是对关键信息保留和语义一致性要求较高的压缩任务。
- GLM-Edge-4B：适用于对首响应时间要求高、允许一定信息损失的快速摘要场景。
- Gemma-4-E2B：适合作为备选方案，用于补充验证不同模型族的表现。
- Gemma-4-E4B：当前硬件条件下不建议作为主力模型。
- Phi-4-mini：适合用于探索极限压缩边界，不建议直接用于正式上线。

## 8. 结论与推荐

### 8.1 最终推荐

综合本次测试结果，建议采用以下模型选型策略：

| 角色 | 推荐模型 | 推荐理由 |
| --- | --- | --- |
| 默认方案 | Qwen3.5-4B-Q4_K_M | 质量表现最优，语义相似度和关键信息召回率整体领先，且性能可接受 |
| 低时延备选 | GLM-Edge-4B | 首 Token 延迟和吞吐率表现最好，适合强调响应速度的场景 |
| 备选验证 | Gemma-4-E2B-it-Q8_0 | 性能与质量较为平衡，可作为横向验证模型 |

### 8.2 不推荐方案说明

- Gemma-4-E4B-it-Q4_K_M：推理耗时显著偏高，在当前测试环境下投入产出比不佳。
- Phi-4-mini-instruct-Q4_K_M：压缩激进但质量下滑明显，不适合需要高保真压缩的业务场景。

## 9. 风险与局限性

本次测试结果可为模型选型提供依据，但仍存在以下边界：

1. 测试样本数量有限，仅包含 4 个测试集，覆盖范围仍可进一步扩展。
2. test_a 为模拟数据，test_b 至 test_d 为真实日志，数据分布存在差异，结论需结合样本来源理解。
3. 关键信息召回率采用 LLM-as-Judge 评估，尽管具备较强可解释性，但仍存在一定模型主观性。
4. 测试环境限定为 Windows 11 + 8GB 显存本地部署，结论不直接等同于更高配置或 Linux 服务器环境下的表现。
5. 当前提示词更偏向总结压缩，未针对“强约束保留关键字段”进行专项优化，因此长文本召回率仍有提升空间。

## 10. 后续优化建议

建议下一阶段围绕以下方向继续优化：

1. 对提示词进行专项优化，增加对关键字段、时间、数值和工具结果的强约束保留要求。
2. 引入更多真实业务日志样本，覆盖不同长度、不同结构和不同业务类型的上下文数据。
3. 在正文摘要压缩之外，探索“结构化压缩”方案，例如按事件、决策、工具结果和待办事项分段输出。
4. 对召回率评估进行人工抽样复核，提升结论可信度。
